# Phase 3 Study Guide — Domain Layer

## 📖 Concepts Learned

### Classes and Properties
- A **class** is a blueprint. An **object** is one instance built from that blueprint. `new Runbook()` builds one object from the `Runbook` blueprint.
- **Properties** are the things a class "has." The full syntax is `public Type Name { get; set; }` — `public` (visible everywhere), the type (what kind of data), the name, and `{ get; set; }` (readable and writable).
- **Default values** can be assigned inline: `public string Author { get; set; } = string.Empty;`. The value is set automatically when the object is created.
- **One type per file** is the C# convention. Each class, enum, and interface gets its own `.cs` file named after the type.

### File-Scoped Namespaces
- `namespace Sentinel.Domain;` with a semicolon (no curly braces) is **file-scoped namespace** syntax introduced in C# 10. Everything in the file belongs to that namespace. It removes one level of indentation from every file.

### `Guid` vs `int` for IDs
- **`int` IDs** are generated by the database — you must insert the record first to know its ID. This creates a dependency between the domain and the database.
- **`Guid` IDs** are generated by the application (`Guid.NewGuid()`) before the database is involved. The domain can create and reference entities without any database call. This is the right choice for Clean Architecture.
- A `Guid` looks like `3f2a1b4c-8e21-4d9a-bb31-2c7f1a9e0d43` — 128 bits of randomness, globally unique.

### Abstract Classes
- `abstract` marks a class as a template only — it cannot be instantiated directly. `new BaseEntity()` is a compile error.
- Abstract classes exist to be extended. They define shared structure without being a complete, usable thing on their own.
- Use `abstract` when a concept is real but incomplete — "every entity has an ID" is real, but "an entity" alone is not a complete thing.

### Inheritance
- `: BaseEntity` is the inheritance syntax. It means *"this class is a BaseEntity — it has everything BaseEntity has, plus its own additions."*
- Inherited properties don't need to be redeclared. `Runbook` gets `Id`, `CreatedAt`, and `UpdatedAt` for free.
- C# supports single inheritance — a class can only extend one other class. It can implement multiple interfaces (covered in a later phase).
- The relationship is always *"is a"*: a `Runbook` is a `BaseEntity`. If the sentence doesn't make sense, inheritance is the wrong tool.

### The `required` Keyword
- `required` is a compile-time guarantee — if you create an object without setting a `required` property, the build fails. It is not a runtime check or a validation rule.
- Use `required` for properties without which the object makes no sense. A `Runbook` without a `Title` is meaningless.
- `required` is different from a constructor parameter — it still allows `{ get; set; }` (mutable after creation), whereas a constructor parameter enforces the value only at creation time.

### Nullable Types — `string?`, `DateTime?`, `float[]?`
- Adding `?` to a type marks it as **nullable** — the value is allowed to be absent (`null`).
- **`string?`** — an optional string. A `Runbook` might not have a `Description`. Use `string?` when the field is genuinely optional.
- **`DateTime?`** — a nullable value type. `ResolvedAt` has no value when an incident opens; it only exists once resolution happens. Without `?`, `DateTime` always has a value (defaulting to `DateTime.MinValue`, which is meaningless noise).
- **`float[]?`** — a nullable array. An embedding doesn't exist at creation time; it gets filled in later by an AI process.
- The `?` pattern communicates intent to every future reader: *"this field might legitimately not exist."*

### Enums
- An **enum** is a fixed set of named constants. The compiler enforces the list — only declared values are valid.
- Under the hood each value is an integer (`Draft = 0`, `Active = 1`, `Deprecated = 2`), but you always use the name, never the number.
- **Type safety:** `RunbookStatus` and `IncidentStatus` are distinct types. You cannot pass one where the other is expected — the compiler catches it.
- Default values on enum properties encode business rules: `= RunbookStatus.Draft` means every runbook starts as a draft until explicitly promoted.

### Navigation Properties
- A **navigation property** is a property whose type is another entity. It lets you traverse from one entity to a related one in code.
- They always appear in pairs — a **foreign key** (the ID stored in the database) and the **navigation property** (the object you traverse to):
  ```csharp
  public Guid? RunbookId { get; set; }   // foreign key — stored in DB column
  public Runbook? Runbook { get; set; }  // navigation — the actual object
  ```
- Both are nullable here because the relationship is optional — an incident might not have an associated runbook.
- EF Core (Phase 4) reads these pairs automatically and generates the correct SQL JOIN. You don't write the JOIN; you navigate the property.

### Arrays and AI Embeddings
- `float[]` is an **array** — an ordered, fixed-size list of floating-point numbers.
- An **AI embedding** is a list of numbers that represents the *meaning* of a piece of text. Similar meanings produce similar numbers. This enables semantic search: finding content by meaning, not by exact keyword match.
- OpenAI's `text-embedding-3-small` model produces 1,536 numbers per piece of text. All 1,536 are stored as a single `float[]`.
- The `pgvector` package (installed in Phase 1) stores `float[]` efficiently in PostgreSQL as a `vector` column and enables fast similarity search. In the Domain layer we use `float[]` — Infrastructure handles the pgvector translation.

## 💡 Interview Q&A

**Q: What is the Domain layer in Clean Architecture and what belongs in it?**
A: The Domain layer contains the core business concepts — entities, enums, value objects, and domain interfaces. It has no dependencies on any framework, database, or external service. It describes *what the system is*, not *how it works*. In .NET it is a plain class library with no NuGet packages. If you can't describe a concept in pure C# without importing anything, it doesn't belong in Domain.

**Q: Why use `Guid` instead of `int` for entity IDs?**
A: `int` IDs are generated by the database — you can't know an entity's ID until after you've inserted it. `Guid` IDs are generated by the application before any database interaction, which keeps the Domain layer independent of the database. It also makes distributed systems easier: two servers can generate IDs simultaneously with no coordination and no collision.

**Q: What is an abstract class and when should you use one?**
A: An abstract class is a class that cannot be instantiated directly — it exists only to be extended. Use it when you have shared structure that makes no sense on its own. `BaseEntity` is abstract because "an entity" with just an ID and timestamps is not a complete concept — only a `Runbook` or `Incident` is complete.

**Q: What is the difference between `string`, `string?`, and `required string`?**
A: `string` is non-nullable — it must always have a value (never null). `string?` is nullable — the value may legitimately be absent. `required string` is non-nullable and must be set at the time the object is created; the compiler enforces this. These three options let you express exactly how optional a field is, directly in the type.

**Q: What is a navigation property and why does it come with a foreign key?**
A: A navigation property is a property on an entity whose type is another entity — it lets you traverse relationships in code (`incident.Runbook.Title`). The companion foreign key (`RunbookId`) is the actual value stored in the database column. EF Core reads both together: the foreign key tells it which row to JOIN, and the navigation property is where it puts the resulting object. You need both because the database stores IDs, but application code works with objects.

**Q: What is a vector embedding and why store it in the database?**
A: An embedding is a list of numbers produced by an AI model that represents the meaning of a piece of text. Texts with similar meanings produce numbers that are close together in mathematical space. Storing embeddings in the database (via pgvector) allows the app to find knowledge entries that are semantically similar to a user's question — not just entries that share exact keywords. This is the foundation of Retrieval-Augmented Generation (RAG), which lets the AI answer questions using your specific documentation.

## 🔗 How It Connects to Other Phases

- **Phase 4 (Database)** maps these entities to PostgreSQL tables using EF Core. `BaseEntity` properties become standard columns. Navigation properties become foreign key constraints. `float[]` becomes a `vector` column via pgvector. The Domain classes don't change — EF Core configuration lives entirely in Infrastructure.
- **Phase 5 (Authentication)** adds an `ApplicationUser` entity (ASP.NET Core Identity). It follows the same patterns: extends a base class, has nullable and required properties.
- **Phase 6 (Application layer)** defines service interfaces that accept and return these Domain entities. `IRunbookService`, `IIncidentService`, `IKnowledgeService` — all typed to the classes built here.
- **Phase 8 (AI Assistant)** reads `KnowledgeEntry.Embedding` values to perform similarity search. The `float[]` property defined here is the exact data the RAG pipeline will query.
- **Phase 2 (UI — already done)** — the `<StatCard Value="0" />` placeholders on the dashboard will eventually display `Count()` queries against these entities.

## 📝 Quick Reference

```csharp
// File-scoped namespace (no braces needed)
namespace Sentinel.Domain;

// Abstract base class
public abstract class BaseEntity
{
    public Guid Id { get; set; } = Guid.NewGuid();
    public DateTime CreatedAt { get; set; } = DateTime.UtcNow;
    public DateTime UpdatedAt { get; set; } = DateTime.UtcNow;
}

// Entity with inheritance
public class Runbook : BaseEntity         // ": BaseClass" = inheritance
{
    public required string Title { get; set; }    // must be set at creation
    public string? Description { get; set; }       // optional (nullable)
    public string Author { get; set; } = string.Empty;  // optional with default
    public RunbookStatus Status { get; set; } = RunbookStatus.Draft;
}

// Enum
public enum RunbookStatus
{
    Draft,         // = 0 under the hood
    Active,        // = 1
    Deprecated     // = 2
}

// Navigation property pair
public Guid? RunbookId { get; set; }    // foreign key stored in DB
public Runbook? Runbook { get; set; }   // navigation to the full object

// Nullable value types
public DateTime? ResolvedAt { get; set; }   // absent until resolved
public float[]? Embedding { get; set; }     // absent until AI processes it

Nullability rules:
  string    → non-nullable reference type (must have value)
  string?   → nullable reference type (can be null)
  DateTime  → non-nullable value type (always has a value, defaults to MinValue)
  DateTime? → nullable value type (can be null)
  float[]   → non-nullable array (must have value)
  float[]?  → nullable array (can be null)
```
